# Архив-панель — фиксы по отчёту тестера (источник целей)

Источник: живая проверка тестера от 2026-06-25 по сборке `cb708e5` (все 4 ТЗ собраны,
гейт+ревью пройдены, но live-проверка вскрыла баги). HTML-отчёт:
`docs/archive-panel-test-report.html`.

Ниже — **5 целей** (каждую можно ставить в `/goal` отдельно), по приоритету.
Сквозные требования для ВСЕХ: гейт (fmt + clippy −D + test ×3 + i18n_guard; cargo-deny
при новых dep) · ревью-агент по диффу 0 critical / 0 important · UI-фикс → живая
проверка тем же тестером по обновлённому HTML-отчёту · публикация только по явному
«релизь», копим в один проверенный релиз.

Отчёт тестера (баги дословно): см. раздел «Сырой отчёт» внизу.

## ЦЕЛЬ 1 — 🔴 Блокеры: плеер играет не туда + краш удаления

Без этих двух панель нельзя релизить.

### F1. Плеер: клик по реплике играет НЕ с её начала (рассинхрон таймкод↔аудио)
- Симптом: клик по реплике → звук с ~следующей реплики, не с первого слова; первая
  реплика показана как `[00:29]`, а не `[00:00]` — всё смещено на оффсет до первой реплики.
- Вероятная причина: `start_ms = utterance.unix_ms − session_start`, но `unix_ms` — момент
  ФИКСАЦИИ реплики (≈её конец). По решению #6 начало строки i = таймштамп предыдущей
  строки (у первой = старт сессии/записи) → seek сейчас попадает в конец реплики.
- Критерии приёмки:
  - [ ] Первая реплика показывает `[00:00]` (или корректный старт записи), не время конца.
  - [ ] Клик по любой реплике → звук с её ПЕРВОГО слова (на слух), без смещения.
  - [ ] Таймкод у реплики == позиция в плеере, где она звучит.
  - [ ] Те же исправленные таймкоды в обычном просмотре стенограммы и в «Копировать с таймкодами».
  - [ ] Старт аудио-микса совпадает со стартом сессии (иначе постоянный сдвиг) — выровнять.
  - [ ] Юнит-тест маппинга «строка i → позиция», покрывает первую строку и паузу до неё.
  - [ ] Проверено тестером на ≥2 созвонах с записью.

### F2. Удаление: КРАШ приложения при клике на 🗑
- Симптом: нажатие корзины → приложение падает.
- Критерии приёмки:
  - [ ] Клик 🗑 → диалог подтверждения, БЕЗ краша.
  - [ ] «Отмена» → ничего не удалено; «Удалить» → сессия исчезает + файлы стёрты, без краша.
  - [ ] Активную сессию удалить нельзя — понятное сообщение, без краша.
  - [ ] Найден и устранён КОРЕНЬ краша (не «поймали панику»); лог чист; повтор идемпотентен.

## ЦЕЛЬ 2 — 🟠 Ретро-резюме: довести до ТЗ3

### F3. «Пересоздать» без подтверждения
- Симптом: при существующем резюме повтор сразу пересоздаёт, без вопроса.
- Критерии: [ ] есть резюме → «Пересоздать» только после подтверждения; отмена → не трогает.

### F4. ↻ активна у пустых сессий, но молчит
- Симптом: у сессии без данных кнопка нажимается, ничего не происходит.
- Критерии: [ ] нет данных → кнопка НЕактивна + подсказка «Недостаточно данных» (ТЗ3).

## ЦЕЛЬ 3 — 🟠 UI-полировка: SVG вместо эмодзи + кейс «Аудио не сохранено»

### F5. Эмодзи вместо SVG (правило проекта)
- Симптом: ▶/⏸ в плеере — эмодзи; чекбокс реплики — просто квадрат без SVG-галочки.
- Критерии:
  - [ ] ▶/⏸ — SVG (переиспользовать иконку старт/пауза из главного бара).
  - [ ] Чекбокс реплики — SVG-галочка, не текстовый «✓».
  - [ ] Светлая и тёмная темы — ок; гейт зелёный.

### F6. «Аудио не сохранено» не показывается в нужном кейсе
- Симптом: у сессии без записи видно «Транскрипт не сохранён», плеера и «Аудио не
  сохранено» нет вовсе (пустой транскрипт → скрыт весь блок).
- Критерии:
  - [ ] «есть транскрипт, нет записи» → «Аудио не сохранено» (плеер скрыт).
  - [ ] «нет транскрипта» → «Транскрипт не сохранён» (как сейчас). Различать корректно.

## ЦЕЛЬ 4 — 🟡 Производительность появления окон

### F7. Окна появляются с задержкой 1–3 с (видна обводка; ~1–3 с не кликается)
- ВАЖНО: тестер гонял DEBUG (`cargo run`) — она в разы медленнее релиза.
- Критерии:
  - [ ] Перепроверить на RELEASE-сборке.
  - [ ] Если на release >~300 мс — диагностировать `present_window_stealth_aware`/reveal-tick
        и сократить; иначе закрыть как debug-артефакт (с замером).

## ЦЕЛЬ 5 — ⚪ (отдельно) Настройки → вкладка «Память» тормозит/ломает приложение

### F8. «Память» открывается очень долго; у одного юзера краш
- Пред-существующее, НЕ связано с архив-панелью; релиз панели не блокирует, но краш у
  юзера критичен сам по себе.
- Критерии:
  - [ ] Найдена причина медленного открытия вкладки «Память».
  - [ ] Воспроизведён и устранён краш у затронутого юзера (или понятая причина + защита).
  - [ ] Открытие вкладки — без заметной задержки.
