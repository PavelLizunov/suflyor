- Теряется при рестарте: транскрипт, last_q/a, session_cost, health, открытые тайлы (`runtime_state.rs`/`app_state.rs`). Журнал+конфиг переживают.

Факты из кода (опора):
- `config.rs:501 ai_endpoint(prep: bool)` — если `ai_provider=="local"` → локалка (и live, и prep); иначе облачный мост. Облачные поля (`ai_base_url/bearer/ai_model/prep_model`) ЛЕЖАТ в конфиге всегда, даже когда активен local.
- `prep_model` (default `claude-sonnet-4-6`, `config.rs:31/406`) — уже настроенная «умная» облачная модель (сейчас для prep/дебрифа). Переиспользовать её как escalation-таргет.
- Web-search/tool-use в `ai.rs` НЕТ (греп `web_search|tools|tavily|perplexity` → пусто).
- `fire_regenerate`/`fire_followup_ask` уже принимают флаг `use_vision` и резолвят endpoint — escalation добавляется тем же паттерном (флаг `force_cloud`).

Building block (бэкенд, маленький + чистый):
- Новый резолвер `config.rs: ai_endpoint_cloud()` — ВСЕГДА облачные поля, ИГНОРИРУЯ `ai_provider`; модель = `prep_model` (умная), bearer/base_url = облачные. Чистое ДОБАВЛЕНИЕ, существующий `ai_endpoint()` не трогаем. + юнит-тесты (как у `ai_endpoint_*`).
- One-shot: постоянная настройка остаётся `local` — возвращать ничего не надо, «забыть вернуть» невозможно. Один вопрос ушёл в cloud, следующий снова локальный.

UX (три триггера, общий бэкенд):
- A. Хоткей «умный ask» (напр. Shift+F9): F9=local как сейчас, Shift+F9=cloud разово. Регистрация рядом с F3/F4/F6/F8/F9 (`overlay_host.rs:1857-1895`). i18n-метка хоткея.

- C. Префикс/тумблер 🧠 в строке follow-up тайла (рядом с 🎤/→): следующий follow-up уходит в cloud. Через `fire_followup_ask(..., force_cloud=true)`.
- Маркер: на эскалированном тайле транзиентный бейдж «🧠 cloud» (видно, что ЭТОТ ответ из облака). Бар-индикатор активного стека остаётся `local` (дефолт не менялся).
