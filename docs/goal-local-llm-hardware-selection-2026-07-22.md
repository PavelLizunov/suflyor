# Выбор локальной LLM по железу — оценка и рекомендация, 2026-07-22

## Короткий вывод

**Да, идею стоит реализовать, но не как перенос или запуск `llm-checker`.** Для
suflyor достаточно маленького нативного советника по двум уже поддержанным
моделям: показать, какая из Gemma 4B / 12B подходит этому ПК, автоматически
предлагать модель только при подтверждённой пригодности и всегда оставить ручное
переключение.

Полный каталог моделей, Ollama, Node.js, автоматический `pull` и универсальный
скоринг в suflyor сейчас не нужны. Они заметно увеличат поверхность отказов, а
реальная пользовательская задача уже почти закрыта существующим переключателем
«быстрее 4B / умнее 12B».

## Какой ресурс найден

Это сильное совпадение с описанием («определить железо и подсказать локальную
LLM»), но открытые профили не доказывают связь `Pavelevich` с `PavelLizunov`.
Если владелец имел в виду другой, приватный или перенесённый репозиторий, для
точного аудита нужна его ссылка или доступ; этот вывод не подменяет такой аудит.

### Проверяемость источника

Проверены публичные README, `package.json` и LICENSE в ветке `main` на
2026-07-22. Это **не побайтовая аттестация** и не доказательство того, что
кандидат является личным репозиторием владельца: в рабочей среде нет полученной
копии исходников, для которой можно честно записать SHA-256. Поэтому документ
не заявляет контрольную сумму внешнего файла и не опирается на неё; вывод —
независимая оценка наблюдаемого функционала и лицензии. Если когда-либо
понадобится заимствование кода, предварительным условием будет отдельная
проверка: зафиксировать commit/tag, получить именно эти байты, локально
вычислить SHA-256 нужных файлов и записать commit вместе с суммой. В этой
задаче код из ресурса не используется.

Что делает `llm-checker`:

- определяет CPU, GPU, RAM/VRAM и доступный backend;
- ранжирует модели по Quality / Speed / Fit / Context;
- выбирает квантование, которое помещается в память;
- работает с каталогом Hugging Face / Ollama / GPT4All и локальными моделями;
- умеет запускать Ollama-модели и измерять скорость ответа;
- строит калиброванную policy для последующего выбора модели.

Это хороший отдельный универсальный CLI. Его сильная идея для suflyor — не
каталог из десятков тысяч артефактов, а честное объяснение «что запустится на
этом компьютере и с какой ожидаемой скоростью».

## Что уже есть в suflyor

Половина нужного пользовательского сценария реализована:

- `local_ai.rs` выбирает CUDA для NVIDIA, Vulkan для AMD/Intel и CPU fallback;
- базовая Gemma E4B устанавливается всегда, Gemma 12B скачивается по запросу;
- в Настройках пользователь вручную переключает «быстрее 4B / умнее 12B»;
- UI уже предупреждает, что 12B примерно вдвое медленнее и требует около
  10 GB VRAM;
- оба веса, размеры и SHA-256 закреплены, а vision-проектор выбирается строго
  под активную модель;
- при **первоначальной установке E4B** неудачный GPU-запуск повторяется на CPU.

### Нерешённая готовность ручного переключения

Ручное переключение пока **не** доказывает готовность модели. `switch_local_model`
считает `is_reachable(/v1/models)` успехом при любом HTTP-ответе: `curl` не
проверяет код и тело. Поэтому 404 или 503 во время загрузки могут дать
`ModelSwitch::Switched`, после чего UI сохранит выбор. Это особенно опасно для
12B: процесс может занять порт, затем завершиться или остаться в 503.

До показа пользователю успешного переключения этот путь должен получить общий
с установщиком контракт готовности: дочерний процесс ещё жив, `GET /v1/models`
вернул 2xx и содержит именно запрошенный GGUF, а короткий `POST
/v1/chat/completions` вернул 2xx и валидный `choices`. 503 при прогреве надо
повторять до deadline; постоянный 503, 404, несовпавшая модель и завершение
процесса должны вернуть `FailedToStart` и не записывать настройку. Для этого
изменения обязательны регрессии с постоянным 503; с несколькими 503 и
последующим успехом; с 404; с 2xx `/v1/models`, в котором указан другой GGUF;
с 2xx completion без `choices` или с `choices` неправильной формы; и с
завершением дочернего процесса во время ожидания. Тесты должны утверждать и
результат переключения, и отсутствие записи нового выбора в конфигурацию.

Чего нет: suflyor определяет **тип** ускорителя, но не его доступную память и не
оценивает пригодность 4B/12B до многогигабайтного скачивания. Поэтому сейчас
одинаковую кнопку «скачать 12B» видит и RTX с большим запасом VRAM, и слабый ПК,
на котором модель уйдёт в медленный CPU/RAM-сценарий.

## Почему нельзя встроить `llm-checker` целиком

1. **Стек.** `llm-checker` — Node.js 16+ и ориентируется на Ollama. suflyor по
   правилам проекта — чистый Rust + Slint и уже управляет собственным
   `llama-server`. Добавлять второй runtime и второй менеджер процессов нельзя.
2. **Несоразмерный масштаб.** Для выбора между двумя проверенными весами не
   нужны SQLite-каталог, синхронизация реестров и четырёхмерный рейтинг.
3. **Надёжность.** Произвольная модель требует проверки chat template,
   thinking-режима, 8192 context, русского/английского интервью, streaming,
   лицензии и совместимого vision-проектора. Совпадение по VRAM этого не
   гарантирует.
4. **Supply chain.** Текущие модели скачиваются с закреплёнными размером и
   SHA-256. Живой каталог и команды установки ослабят этот инвариант.
5. **Лицензия.** `llm-checker` опубликован под NPDL-1.0 с запретом платной
   дистрибуции/сервиса. Это дополнительное ограничение несовместимо с прямым
   переносом производного кода в GPL-3.0-проект без отдельного разрешения
   правообладателя. Идею можно реализовать независимо, код копировать нельзя.

## Рекомендуемый маленький MVP

Реализовывать отдельной задачей, не в рамках этого исследовательского решения.

### 1. Нативный профиль железа

В `overlay-backend` добавить чистую структуру с:

- total/free system RAM;
- отдельной записью для **каждого** адаптера: стабильный id, имя, возможный
  backend, dedicated total/free VRAM и признак shared/unified memory;
- явно выбранным `execution_adapter` и причиной выбора;
- причиной результата и `Unknown`, а не догадку, когда память определить нельзя.

Политика размещения должна быть явной и одинаковой у советника и запуска
`llama-server`: первый MVP рассматривает только один выбранный dedicated GPU и
**никогда не складывает** VRAM нескольких карт или dedicated VRAM с shared RAM
iGPU. Рекомендовать 12B можно лишь после того, как выбранный адаптер передан
рантайму как целевой; пока такого выбора нет, результат — `unknown` / «нужна
проба запуска», а не рекомендация по памяти другой карты. Поддержку настоящего
multi-GPU можно добавить позднее, только вместе с явной политикой placement в
`llama.cpp` и отдельной проверкой этой конфигурации.

Для NVIDIA запрашивать в CSV `index,uuid,pci.bus_id,name,memory.total,memory.free`
**по каждой карте в момент оценки**, чтобы занятая VRAM не выглядела свободной.
`uuid` — обязательный сохранённый идентификатор выбора, `pci.bus_id` — его
диагностическая проверка; name и index только отображаются и никогда не
сохраняются как ключ. Индекс NVIDIA не стабилен после перезагрузки, а имя может
повторяться — NVIDIA рекомендует UUID или PCI bus ID для постоянного выбора.

При каждом запуске сохранить/разрешить именно UUID: дочернему `llama-server`
передать `CUDA_VISIBLE_DEVICES=<полный UUID>`, с тем же окружением вызвать
`--list-devices`, взять единственное имя CUDA из вывода и запустить
`--device <это-имя> --split-mode none --main-gpu 0`. Так `--device` выбирает
единственное видимое устройство, а не меняющийся глобальный ordinal. Если UUID
исчез, PCI bus ID не совпал или список не даёт ровно одного CUDA-устройства,
результат — `Unknown`, без рекомендации по этой карте. Для AMD/Intel нельзя
считать `Win32_VideoController.AdapterRAM`
абсолютной истиной: iGPU использует shared RAM, а старые WMI-поля обрезают большие
значения. На таких системах безопаснее показать «нужна проба запуска», чем
ложное «подходит».

### 2. Детерминированная оценка только поддерживаемых моделей

Не переносить Q/S/F/C-скоринг. Хранить рядом с закреплёнными моделями небольшой
manifest:

- размер файла и оценка рабочей памяти с context 8192;
- поддерживаемые backend/vision/chat-template;
- `recommended`, `borderline`, `not_recommended` или `unknown`;
- короткое объяснение результата.

Числовые пороги в manifest пока добавлять нельзя: для текущей 4B минимальный
набор артефактов уже занимает 6 570,47 MiB (GGUF 4 746,60 MiB + обязательный
F32-проектор 1 823,88 MiB) ещё до context, KV-cache и буферов runtime. У 12B
GGUF занимает 6 405,22 MiB, а с её F16-проектором — 6 572,22 MiB. Размеры файлов
не равны фактически занятой VRAM, но доказывают, что прежние 5 120/6 144 MiB для
4B не могут быть ни working-, ни recommended-порогом. Их нельзя использовать
как рекомендацию или тестовую границу.

До калибровки советник может показывать состав железа и `Unknown` с причиной
«для этой конфигурации ещё нет измеренного профиля», но не имеет права
автоматически рекомендовать, скачивать или активировать модель. Для каждой
поддерживаемой конфигурации нужны отдельные измерения: 4B + F32-проектор, 12B
text-only и 12B + F16-проектор. Замер должен использовать закреплённые GGUF,
текущий build `llama-server`, тот же backend и placement, `-c 8192`, включённый
GPU-offload проектора и короткий chat-запрос; vision-запрос обязателен только
для профиля с проектором. Он записывает пик
занятой памяти выбранного устройства после прогрева, KV-cache и runtime-буферы,
а также минимальную доступную system RAM. Только после повторяемого замера на
Windows для каждого backend можно закрепить двоичные MiB: working — доказанная
граница успешного полного профиля, recommended — working плюс заранее
обоснованный запас. Эти данные и версия runtime входят в manifest; при их смене
профиль снова становится `Unknown`.

Правило имеет ровно четыре результата:

1. `Recommended(model)` — выбранный dedicated GPU имеет измеренный в manifest
   free-VRAM порог, требуемую manifest system RAM, а manifest подтверждает
   backend и точный профиль модели. Если обе модели проходят, выбирается 12B:
   только среди полноценно подтверждённых GPU-профилей качество важнее скорости.
2. `Borderline(model)` — только ручная проба: либо VRAM покрывает измеренный
   working-порог, но не запас recommended, либо CPU-путь модели проходит его
   измеренный порог доступной RAM. Если несколько вариантов только borderline,
   результат всегда `Borderline(4B)`: при отсутствии подтверждённой GPU-рекомендации
   приоритет у скорости и меньшего риска. Это включает CPU-only ПК, на котором
   формально помещаются и 4B, и 12B; 12B можно перечислить как ручную
   альтернативу, но она не меняет единственный результат.
3. `NotRecommended` — все нужные измерения получены, но ни один manifest не
   проходит даже Borderline, либо для текущего backend вообще нет поддерживаемой
   модели. При первой установке local AI остаётся неустановленным и неактивным:
   ничего не скачивается и не выбирается автоматически. Явное действие
   пользователя «скачать и попробовать 4B» может обойти только этот совет, но
   должно показать предупреждение о непроверенном запуске и никогда не называется
   безопасным вариантом.
4. `Unknown` — нельзя надёжно вычислить необходимый путь: нет available system
   RAM, нет UUID/free VRAM выбранного dedicated GPU, выбранный адаптер неоднозначен
   или это iGPU/shared memory без достоверного измерения; сюда же относится
   отсутствие измеренного профиля. Это **не** рекомендация 4B: local AI не
   скачивается и не активируется автоматически. Кнопка явной ручной проверки
   остаётся доступной, но текст честно сообщает, что запуск не предсказан.

VRAM нескольких карт и shared RAM iGPU по-прежнему не суммируются. Неизвестное
железо не блокирует ручную загрузку, но 12B не будет самопроизвольно
перезапущена на CPU.

Чистые unit-тесты правила обязательны как минимум для:

- отсутствия измеренного профиля: `Unknown` и отсутствие автоматической
  загрузки/активации;
- для каждого внесённого измеренного профиля — free VRAM и available RAM ровно
  на working- и recommended-границе и на 1 MiB ниже каждой из них;
- GPU, у которого 12B достигает только `GPU working`, но не recommended-запаса
  (12B `Borderline`, при достаточных 4B — итоговая рекомендация 4B);
- CPU-only ПК на измеренной границе 4B и на 1 MiB ниже неё; CPU-only ПК, где
  проходят обе модели, обязан вернуть `Borderline(4B)`, а не недетерминированный
  результат;
- отсутствие поддерживаемого manifest (`NotRecommended`) и каждую отсутствующую
  обязательную метрику (`Unknown`);
- гибридный ПК (дискретная карта плюс iGPU/shared RAM), две GPU, чью память
  нельзя складывать, и выбранный адаптер, отличный от самого большого;
- перестановку двух одинаково названных NVIDIA-карт: сохранённый UUID должен
  выбрать ту же карту после смены index; разные UUID/PCI должны не смешиваться;
  пропавший UUID и неединственный `--list-devices` должны давать `Unknown`.

Эти тесты проверяют решение, причину и сформированные environment/args, а не
WMI или настоящий `nvidia-smi`.

### 3. Одна подсказка в существующем UI

Под текущими кнопками показать, например:

> Для этого ПК рекомендуется 12B: на выбранном GPU найдено 16 GB свободной
> VRAM; измеренный профиль для этой конфигурации проходит с запасом.

или:

> Не удалось оценить запуск: объём видеопамяти выбранного GPU не подтверждён.
> 4B остаётся доступной для ручной проверки, но не рекомендована автоматически.

или, для измеренного слабого ПК:

> Ни одна поддерживаемая модель не рекомендована: доступной RAM меньше
> измеренной границы даже для 4B. Можно вручную скачать и попробовать 4B.

При `Recommended(4B)` приложение может предложить загрузить 4B, а при
`Recommended(12B)` — предложить 12B; без явного действия пользователя оно ничего
не скачивает и не переключает. При `NotRecommended` и `Unknown` нет активного
«безопасного» дефолта. После ручного выбора suflyor не должен сам переключать
модель ни при старте, ни между запросами.

### 4. Измерение вместо обещаний

После установки можно добавить необязательную кнопку «Проверить на этом ПК»:
один короткий фиксированный prompt через уже запущенный `llama-server`, с
измерением времени до первого токена и tokens/sec. Результат информирует
пользователя, но не меняет модель автоматически. Нельзя скачивать 6 GB только
ради benchmark.

## Когда расширять список моделей

Третью и последующие модели добавлять только если suflyor-специфический тест
покажет заметный выигрыш над Gemma 4B/12B. Для каждого кандидата обязательны:

- закреплённые URL, размер и SHA-256;
- проверенная лицензия распространения;
- тесты короткого RU/EN ответа, кода, follow-up и 8192 context;
- совместимость с текущим llama.cpp и streaming API;
- отдельное решение по vision, без молчаливой потери F8;
- живой замер latency на целевом слабом и сильном железе.

Универсальный каталог остаётся инструментом исследования для разработчика, а не
частью пользовательского приложения.

## Решение

- **SHIP позже:** после Windows-замеров полный pure-Rust hardware-fit советник
  для нынешних 4B/12B с ручным override и, опционально, benchmark уже
  установленной модели. До этого допустим только информационный профиль
  железа без автоматического выбора.
- **DO NOT SHIP:** Node/Ollama-зависимость, код `llm-checker`, динамический
  каталог, автоматическую загрузку произвольных моделей и переключение модели
  без действия пользователя.
- **Сейчас код не менять:** сначала согласовать этот узкий UX; он не исправляет
  баг и не должен ехать без Windows-проверки определения VRAM на NVIDIA и
  AMD/Intel.
