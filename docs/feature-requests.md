## Фича 1 — Кнопка «Копировать» на тайле ответа · **S**

**Сценарий.** Пользователь скриншотит задачу (Notion/Kaiten) → бот распознаёт и
выводит текст в тайле → пользователь хочет забрать текст и вставить дальше, но
кнопки нет, приходится выделять вручную.

**Что нужно.** Иконка 📋 рядом с каждым ответом бота; по клику — весь текст
ответа в буфер обмена; короткий фидбек («Скопировано» / смена иконки).

**Текущий код (точки касания).**
- `ui/tile.slint` — ряд «хрома» уже содержит кнопки 🔄 (`regen-touch`) и
  🧠 (`esc-touch`) как `Rectangle` + `TouchArea` + callback. 📋 встаёт туда же
  той же формой.
- Текст ответа: `OverlayBarBridge.conversations[convo_id]` хранит `messages`
  (Vec<ChatMessage>) и `rendered` (markdown-тред). Чистый источник для копии —
  последний `assistant`-месседж (raw markdown), без HTML/рендера.
- Буфер: app — только Windows. `arboard` (кроссплатформенный, `Clipboard::new()?
  .set_text(s)?`) либо напрямую `clipboard-win` (уже в дереве транзитивно через
  winit). Рекомендую `arboard` — простой API, на Windows под капотом тот же
  `clipboard-win`.

**Набросок реализации.**
1. `tile.slint`: добавить `copy-clicked()` callback + `copied: bool` property;
   новый `Rectangle`+`TouchArea` (📋) слева от 🔄; при `copied` — глиф меняется
   на фидбек-иконку.
2. `overlay_host.rs`: при создании тайла сохранить raw-текст ответа (или читать
   из `conversations` по `convo_id` в момент клика); в обработчике `copy-clicked`
   — `arboard::Clipboard::new()?.set_text(answer)`, затем выставить `copied=true`
   и сбросить через `Timer::single_shot(~1.5s)`.
3. `Cargo.toml`: `arboard = { version = "3", default-features = false }`.

**Нюансы / риски.**
- **Глиф фидбека.** В проекте уже обожглись: `✕`/`⟳` рендерятся «квадратом» на
  Slint+skia. Бэдж «скопировано» НЕ делать голым `✓` (U+2713, тоже text-глиф,
  риск квадрата) — взять эмодзи (✅) либо короткий текст «✓ Скопировано» эмодзи-
  шрифтом / тост. Проверить рендером (PrintWindow) до релиза.
- **Скрин-шеринг.** Копирование локальное, egress нет — безопасно. Сам тайл уже
  уважает стелс (WDA), новых окон фича не создаёт.
- **i18n.** Новые строки («Копировать», «Скопировано») — `@tr` + `ru.po`.
- **Что копировать.** Для типового Q→A — текст ответа (assistant). Для
  многоходового треда решить: только последний ответ или весь тред. Базово —
  последний ответ; полный тред можно дать вторым жестом позже.

**Гейты.** UI-изменение → полный цикл 1–5 (clippy/test/fmt/review/живой смоук +
PrintWindow на бейдж).

## Фича 2 — Текстовый режим «Написать» · **M**

**Сценарий.** Говорить нельзя (шум / людно / тишина). Пользователь жмёт
«Написать» → маленькое окно с полем ввода → печатает вопрос → ассистент отвечает
как обычно. Дополняет голосовой режим, не заменяет.

**Что нужно.** Кнопка вызова текстового ввода (на баре, рядом с микрофоном или
отдельно); маленькое окно с полем + кнопка отправки; ответ — в стандартном
тайле; не мешает основному интерфейсу.

**Текущий код (точки касания).**
- Уже есть наполовину: тайлы умеют ПРОДОЛЖАТЬ диалог печатно —
  `followup-text` ↔ `LineEdit`, `followup-submitted(string)`, `do-submit()`,
  `fire_followup_ask(...)`. То есть печатный ввод вопроса в существующий тайл
  работает.
- Чего НЕТ: входа «задать НОВЫЙ вопрос печатно с бара». Есть голосовые F9/PTT,
  F8 vision, ручной спавн (F6/`manual_spawn_tile`) и палитра F4 (поле ввода —
  но это поиск по KB, не вопрос ассистенту).
- Ask-машинерия для нового тайла: `fire_f9_ask` (берёт транскрипт) и ручной ask.
  Для печатного вопроса нужно подать ВВЕДЁННЫЙ текст как user-prompt (не
  транскрипт) и отрисовать ответ в новый тайл.

**Набросок реализации (вариант A — переиспользовать палитру/мини-окно).**
1. Бар: чип «✏ Написать» (или иконка рядом со стеком капче). Глиф — эмодзи
   (✏/⌨), не text-глиф.
2. Мини-окно ввода: либо новое маленькое Slint-окно (как `palette.slint`,
   frameless + один `LineEdit` + кнопка «Отправить»), либо расширить палитру
   режимом «вопрос ассистенту».
3. На submit: собрать запрос с введённым текстом как user-вопрос (по образцу
   `fire_followup_ask`, но в НОВЫЙ тайл/convo через ask+tile путь как у F9) →
   ответ стримится в стандартный тайл. Точные функции уточню при реализации
   (вероятно тонкая обёртка над общим ask-путём с явным текстом вопроса).

**Нюансы / риски.**
- **Стелс/скрин-шеринг (важно).** Окно ввода — НОВОЕ окно. Под стелсом обязано
  быть исключено из захвата (WDA) и не мигать — применить паттерн
  `present_window_stealth_aware` / park-before-show, как для палитры/Settings.
  Иначе на трансляции мелькнёт окно с печатаемым вопросом.
- **Фокус.** Окно ввода должно забирать фокус клавиатуры (как capture-overlay
  через `focus_window`), но мягко вернуть его после отправки/Esc, не ломая
  always-on-top бара.
- **i18n.** «Написать», «Отправить», плейсхолдер поля — `@tr` + `ru.po`.
- **Не мешать.** Закрытие по Esc/после отправки; не перехватывать глобальные
  хоткеи во время ввода.
- **Хоткей.** Желателен глобальный хоткей-дубль (как у F9/F8) для вызова без
  клика по бару — например, отдельная F-клавиша.

**Гейты.** Новое окно + ввод + ask-путь → полный цикл 1–5, обязательно живой
смоук под стелсом (Win32-проверка affinity=0x11 на окне ввода) и проверка, что
ответ корректно уходит в тайл.

## Фича 3 — Режим перевода для игр (выделил текст → перевод, без описания) · **M**

**Сценарий.** На экране игры — английский текст (субтитры, диалоги, описания
предметов). Возлюбленная хочет быстрый перевод на лету, но F8-vision сейчас
ОПИСЫВАЕТ весь экран («на экране диалог из игры…») вместо того, чтобы просто
перевести выделенный текст. В играх описание экрана — это шум, оно мешает.

**Корень проблемы (заземлено в коде).**
- `overlay-backend/src/vision.rs:17` — единственный фиксированный
  `DEFAULT_VISION_PROMPT` буквально велит модели: «Если это просто экран или
  текст — **кратко опиши суть**». Отсюда «описание вместо перевода».
- `slint-experiment/src/bin/overlay_host.rs:4799-4800` — путь F8 жёстко
  подставляет `DEFAULT_VISION_PROMPT`; способа сказать «это текст, переведи» нет.
- **НО** `vision.rs:26 build_vision_request(image_data_url, prompt)` уже
  принимает ПРОИЗВОЛЬНЫЙ промпт (на дефолт падает только при пустом). Значит
  бэкенд уже готов отправить промпт-перевод — не хватает лишь выбора намерения в
  UI и самого текста промпта.

**Хорошая новость — «выделение текста» уже есть.**
- `slint-experiment/ui/capture_overlay.slint` (V3, Lightshot) уже умеет
  **drag-выделение конкретной области** под курсором: callback
  `region-selected(...)`, подпись «Drag to select a region · Esc / right-click to
  cancel», хоткей **F8** (`help.slint:176`, `settings_panel.slint:632`). То есть
  «выделить субтитры/диалог» — РЕАЛИЗОВАНО. Нужно лишь сказать модели «переведи
  этот кроп», а не «опиши».

**Что нужно (3 части, выбрать комбинацию).**
1. **Промпт-пресет «Перевести».** Рядом с `DEFAULT_VISION_PROMPT` в `vision.rs`
   добавить, напр.: `TRANSLATE_VISION_PROMPT = "Переведи на русский весь текст с
   изображения. Выведи ТОЛЬКО перевод, построчно, БЕЗ описания картинки и БЕЗ
   комментариев. Если текста нет — одной строкой сообщи об этом."`
2. **Способ выбрать режим** до/во время захвата (варианты):
   - **(a)** отдельный хоткей «перевод-захват» — тот же F8-region-путь, но с
     translate-промптом. Самый быстрый «на лету» для игр.
   - **(b)** переключатель режима на самом `capture_overlay`
     («📖 Опиши / 🌐 Переведи»), чтобы один поток покрывал оба намерения.
   - **(c)** меню пресетов (это пункт V5 «config-driven prompt presets» из
     `docs/screen-capture-vision-plan.md` — прочитай/реши/объясни/**переведи**).
   - *Рекомендация:* (a) для скорости + (b) для обнаружимости.
3. **Прокинуть выбранный промпт** в существующую точку отправки
   (`overlay_host.rs:4799` — заменить хардкод константы на выбранный промпт; всё
   дальше — sink/тайл/cost/journal — без изменений).

**Почему это важно.** В играх нужен быстрый перевод; ассистент сейчас
неправильно угадывает намерение (выделил текст ≠ «опиши мой экран»). Явный режим
«Перевести» убирает угадывание.

**Нюансы / риски.**
- **Vision-эндпоинт должен быть настроен** (`vision_provider != off`). Качество
  перевода зависит от vision-модели; для коротких субтитров хватит скромной.
- **Язык-цель** = RU по умолчанию (UI ru/en). Можно читать `ui_language` для
  выбора цели или добавить настройку; v1 — всегда →RU.
- **Дисциплина вывода.** Промпт ОБЯЗАН запрещать описание («только перевод»),
  иначе модель снова начнёт комментировать картинку — в этом вся суть фичи.
- **Egress.** При `vision_provider = cloud` кроп уходит в облако — предупреждение
  как и для обычного захвата (см. screen-capture-vision-plan §7).
- **i18n.** Новые строки (метки режима, подсказка хоткея) — `@tr` + `ru.po`.
- **Стелс.** Нового окна нет (переиспользуем `capture_overlay`, уже под WDA).
- **Связь с Фичей 4.** Translate-пресет — естественное место, чтобы опционально
  добавить фонетическую транскрипцию.

**Гейты.** Трогает vision-промпт (логика) + захват/хоткей (UI). Если добавляем
новый хоткей/тоггл на оверлее → полный цикл 1–5 (+ живой смоук: F8-region по
английскому тексту → тайл показывает ТОЛЬКО RU-перевод). Чисто логический минимум
(2-й хардкод-хоткей с translate-промптом, без UI) → 1–4 + смоук, размер **S**.

## Фича 4 — Транскрипция при переводе (фонетика) · **S–M**

**Сценарий.** При переводе слова/фразы (особенно в игровом режиме перевода,
Фича 3) показывать рядом, КАК слово читается: `schedule [ˈʃedjuːl]`. Полезно для
сложных английских слов с неочевидным произношением.

**Что нужно.** Рядом с переводом — транскрипция МФА. Опционально:
включать/выключать в настройках (по умолчанию — выкл, чтобы вывод оставался
чистым).

**Заземление.**
- В первую очередь это **добавка к промпту** поверх translate-пресета Фичи 3 (в
  `vision.rs`): «…и для каждого нетривиального слова дай транскрипцию в
  квадратных скобках МФА, напр. schedule [ˈʃedjuːl]». Когда/если появится
  текстовый перевод — та же добавка в системный промпт текстового пути.
- Флаг-тоггл (по образцу существующих bool-настроек вроде `ai_local_vision`,
  `config.rs:65`) гейтит фичу; в UI — DarkCheck в `settings_panel.slint` (клон
  существующей строки чекбокса) + `@tr` + `ru.po`.

**Нюансы / риски.**
- **Глифы МФА** (ˈʃedjuːl) — не-ASCII. Известная болячка проекта про «квадраты»
  касается СИМВОЛЬНЫХ глифов (✓ ✕ ⟳) на Slint+skia; буквы МФА — обычный Unicode и
  обычно рисуются нормально, но **проверить через PrintWindow** до релиза.
- **Объём.** Транскрипция осмысленна для слов/коротких фраз; для длинных абзацев
  — шум. Промпт должен транскрибировать только «нетривиальные» слова либо только
  когда переведённая единица ≤ N слов.
- **Зависимость от модели.** Точность МФА зависит от модели; слабая локальная
  может выдумывать транскрипцию. Отметить это; возможно — гейт на сильную/облачную
  модель или пометка «приблизительно».
- **Опциональность.** Дефолт OFF — это power-фича за тогглом.

**Гейты.** Промпт + тоггл-настройка. 1–4 + живой смоук; если тоггл добавляет UI —
полный 1–5 + проверка PrintWindow на рендер МФА в тайле. Размер **S**, если просто
дописать в translate-промпт под фиксированным тогглом; **M** с полноценной строкой
настройки и покрытием текстового пути.

## Баги с окнами (П1 от 2026-06-02) — кластер, нужен repro · **?**

Возлюбленная назвала пункт 1 как «Баги с окнами (закрытие, позиция, блокировка,
количество)» — это КЛАСТЕР без конкретных шагов. Фиксирую четыре симптома с
указанием релевантных поверхностей кода; **каждый требует конкретного repro,
прежде чем станет actionable** (какое окно, какое действие, что ожидалось vs что
произошло). По методологии — не чинить вслепую: сначала repro/откат, без
«волн фиксов».

- **Закрытие.** Тайлы закрываются через `tiles.retain(...)` в обработчиках
  (`overlay_host.rs:2144 / 3047 / 3846 / 4224 / 4675 / 6354`) + чип «закрыть всё»
  (#110, `:2890-2909`). Возможные жалобы: тайл не закрывается, «закрыть всё»
  оставляет «призрак», Esc не закрывает. → нужно: какое окно, как закрывали, что
  осталось.
- **Позиция.** Тайлы ставятся через `win32::pick_monitor` + стекинг; бар при
  старте пинится на primary. У пользователя портретный вторичный монитор при
  x=-1200 — геометрия чувствительна. Возможно: тайл вылезает за экран / не на тот
  монитор / перекрытие. → нужно: раскладка мониторов на момент бага + где
  появилось vs где ожидалось.
- **Блокировка.** Неоднозначно: (a) окно крадёт/блокирует фокус либо
  always-on-top перекрывает нужное; (b) заблокировано взаимодействие; (c)
  стелс/affinity делает окно некликабельным. → нужно: что именно заблокировано и
  каким окном.
- **Количество.** Счётчик тайлов трекается и показывается как «+ tile (N)» с
  пересчётом (`:5416`) после каждого push/retain; спавн рейт-лимитирован
  (`MAX_TILES_PER_MIN_AGGRESSIVE` 10–20/мин, `:2051 / :3246`). Возможно: тайлы
  копятся, неверная цифра в лейбле, неожиданный рейт-лимит. → нужно: сколько окон,
  от какого триггера.

**Статус: заблокировано на repro.** Запросить у Возлюбленной: раскладку
мониторов, точные шаги, какое окно, ожидание vs факт, в идеале — короткий клип.
После этого каждый симптом превращается в отдельный заземлённый баг-репорт.

## Сквозные заметки

- **i18n RU+EN.** Любая видимая строка — `@tr("English…")` в `.slint` +
  пара `msgid/msgstr` в `slint-replay.po`. Хардкод кириллицы без `@tr` = баг.
- **Скрин-шеринг — главный инвариант.** Любое НОВОЕ окно (ввод текста) обязано
  уважать стелс (WDA) и park-before-show. Буфер обмена — локальный, безопасен.
- **Глифы.** Только эмодзи для иконок (📋 ✏ ⌨ ✅); голые text-глифы (`✓ ✕ ⟳`)
  рендерятся квадратом на Slint+skia — проверять PrintWindow. Буквы МФА
  (`ˈʃedjuːl`, Фича 4) — обычный Unicode, но тоже проверить рендером.
- **Vision: «опиши» ≠ «переведи».** Захват региона уже есть (`capture_overlay`,
  F8); намерение задаётся ПРОМПТОМ (`vision.rs`). Перевод (Фича 3) и транскрипция
  (Фича 4) — это варианты промпта поверх готового канала, не новый pipeline.
- **Методология.** Любая фича, трогающая UI/геометрию/прозрачность → полный цикл
