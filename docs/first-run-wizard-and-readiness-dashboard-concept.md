# First-Run Wizard and Readiness Dashboard Concept

Идея: добавить два связанных сценария:

1. **Мастер первого запуска** — помогает быстро настроить приложение на новом компьютере.
2. **Экран диагностики готовности** — в любой момент отвечает на вопрос: «Можно начинать интервью или что-то сломано?»

Это продуктовый концепт для обсуждения. Не инструкция к немедленной реализации.

## Зачем это нужно

У приложения уже много возможностей: cloud/local AI, несколько STT-провайдеров, микрофон, системный звук, stealth, локальные серверы, модели и ключи. Но пользователю приходится самостоятельно понимать зависимости между ними.

Перед интервью важен не полный доступ ко всем настройкам, а уверенность:

```text
AI отвечает
STT распознаёт речь
микрофон слышно
системный звук захватывается
overlay отображается
stealth работает так, как ожидается
```

Мастер помогает собрать рабочую конфигурацию. Диагностика помогает проверить её повторно после переноса на другой ПК, смены гарнитуры, обновления приложения или локальной модели.

## 1. Мастер первого запуска

### Когда показывать

- При первом запуске, если конфиг ещё не создан.
- После импорта server settings на другом компьютере — предложить короткую повторную проверку.
- В Settings оставить кнопку `Запустить мастер настройки`.
- Не блокировать приложение навсегда: мастер можно пропустить и открыть позже.

### Принцип

Один экран — один понятный вопрос. На каждом шаге есть:

- короткое описание;
- основной выбор;
- кнопка проверки, если она нужна;
- явный статус: `OK`, `Требует внимания`, `Пропущено`;
- переход дальше даже при ошибке, но с предупреждением.

### Шаг 1. Режим работы

Предложить три режима:

```text
Cloud
Groq STT + Claude bridge. Быстрый старт, нужны ключи и интернет.

Fully local
Локальный AI + локальный STT. Данные не уходят наружу, нужны модели.

Mixed
Например: локальный STT + cloud AI. Настроить вручную.
```

Внизу показывать итог простым языком:

```text
Сейчас: локальное распознавание + облачные ответы.
Аудио не отправляется в облачный STT. Текст вопроса отправляется AI bridge.
```

### Шаг 2. AI

Для cloud:

```text
AI bridge URL
Bearer token
Model
[Проверить AI]
```

Для local:

```text
Local AI URL
Bearer token, если нужен
Model
[Найти модели]
[Проверить AI]
```

Результат:

```text
OK · claude-haiku-4-5 · ответ за 540 ms
```

или:

```text
Ошибка авторизации · проверь bearer token
```

### Шаг 3. Распознавание речи

Предложить:

```text
Cloud Groq Whisper
Local GigaAM-v3
Local Whisper
```

Показывать только релевантные поля выбранного режима.

Для проверки:

```text
[Записать тест 5 секунд]
```

После теста показать распознанный текст и время:

```text
OK · Local GigaAM-v3 · GPU · 0.8 s
Распознано: «проверка микрофона один два три»
```

### Шаг 4. Микрофон

```text
Устройство: [Headset Microphone ...]
Уровень: ███████░░░
[Записать тест]
```

После записи:

```text
OK · сигнал стабильный · -21 dB
```

Если тишина:

```text
Сигнал не обнаружен. Проверь выбранное устройство и разрешения Windows.
```

### Шаг 5. Системный звук

Нужен для распознавания речи собеседника:

```text
Устройство вывода: [Speakers / Headset ...]
[Проверить системный звук]
```

Подсказка:

```text
Включи короткое видео или любой звук на компьютере, затем нажми Проверить.
```

Результат:

```text
OK · loopback signal detected · -17 dB
```

### Шаг 6. Overlay и stealth

Показать тестовый тайл:

```text
[Показать тестовый тайл]
```

Спросить:

```text
Тайл виден поверх остальных окон?
```

Для stealth:

```text
Скрывать overlay от захвата экрана: [toggle]
```

Можно предложить короткую инструкцию:

```text
Сделай Print Screen или открой предпросмотр screen-share.
Overlay должен исчезнуть из захвата, но оставаться видимым на твоём экране.
```

Важно: не заявлять автоматически, что stealth точно работает, если приложение не может надёжно проверить это программно.

### Шаг 7. Готово

Итоговая таблица:

```text
AI              OK · Cloud · claude-haiku-4-5
STT             OK · Local GigaAM-v3 · GPU
Microphone      OK · Headset Microphone
System audio    OK · Speakers loopback
Overlay         OK · test tile shown
Stealth         ON · verify with screen-share preview
```

Кнопки:

```text
[Начать работу]
[Открыть диагностику]
[Вернуться к проблемным шагам]
```

## 2. Экран диагностики готовности

### Где открыть

- Кнопка в Settings: `Диагностика`.
- Желательно короткая кнопка или иконка на overlay bar.
- После импорта серверных настроек: `Проверить готовность`.

### Основной экран

Не превращать его в лог-файл. Первая секция должна быть короткой:

```text
Готовность к сессии

AI bridge       OK       Claude Haiku · 420 ms
STT             OK       Local GigaAM · GPU
Microphone      OK       Headset Mic · -21 dB
System audio    OK       Speakers loopback · -17 dB
Stealth         ON       ручная проверка screen-share рекомендована
Local servers   OK       llama :8080 · whisper :8081

[Проверить всё]
```

### Статусы

Использовать три уровня:

```text
OK
Требует внимания
Ошибка
```

Опционально:

```text
Не используется
```

Например, если выбран cloud STT, локальный Whisper не должен выглядеть как ошибка:

```text
Local Whisper   Не используется
```

### Детали по клику

Каждая строка раскрывается:

```text
AI bridge
Provider: cloud
URL: http://127.0.0.1:18902/v1
Model: claude-haiku-4-5
Bearer: configured
Last check: OK · 420 ms

[Проверить снова]
[Открыть настройки AI]
```

Не показывать полный bearer token.

### Кнопка «Проверить всё»

Порядок проверки:

1. Проверить AI endpoint.
2. Проверить выбранный STT backend.
3. Записать короткий mic sample.
4. Проверить наличие системного аудиосигнала.
5. Проверить локальные серверы, только если они используются.
6. Показать состояние stealth toggle и предложить ручную проверку.

Не обязательно запускать все проверки строго последовательно. Но UI должен показывать прогресс:

```text
Проверка 3 из 6 · микрофон...
```

### Итог диагностики

Успешный результат:

```text
Готово к сессии
Все используемые компоненты отвечают.
```

Частичная проблема:

```text
Можно продолжить, но есть замечания
Системный звук не обнаружен. Ответы по речи собеседника могут не работать.
```

Критическая проблема:

```text
Не готово к сессии
AI bridge не отвечает. Тайлы с ответами не будут создаваться.
```

### Экспорт диагностики

Добавить:

```text
[Скопировать отчёт]
```

В отчёте:

- версия приложения;
- выбранные providers/models;
- URL без секретов;
- результат проверок и latency;
- названия audio devices;
- GPU/CPU режим;
- timestamp;
- никакого полного bearer/API key.

Пример:

```text
suflyor readiness report
App: 0.4.1
Checked: 2026-05-30 15:40

AI: OK · cloud · claude-haiku-4-5 · 420 ms
STT: OK · local gigaam · GPU · 0.8 s
Mic: OK · Headset Microphone · -21 dB
System audio: WARN · no signal detected
Stealth: ON · manual verification required
```

## Связь с переносом на другой ПК

После `Import server settings...` предложить:

```text
Серверные настройки импортированы.
Локальные устройства и пути могут отличаться на этом компьютере.

[Проверить готовность]
[Открыть мастер настройки]
```

Особенно важно повторно проверить:

- `mic_device`;
- `system_audio_device`;
- `stt_gigaam_dir`;
- локальные AI/STT серверы;
- stealth;
- DPI и позиционирование overlay на новом мониторе.

## MVP

Для первой версии не нужен сложный wizard framework.

Достаточный MVP:

1. Отдельное окно или Settings-панель `Диагностика`.
2. Строки AI, STT, microphone, system audio, stealth.
3. Кнопка `Проверить всё`.
4. Понятные статусы и переходы к нужным вкладкам Settings.
5. В первом запуске показать компактный последовательный flow на основе тех же проверок.

Переиспользовать существующие test handlers, где возможно:

- AI bridge test;
- STT test;
- mic test;
- local server checks;
- текущие config values.

---

## Что важно не сделать

- Не превращать первый запуск в обязательную длинную анкету.
- Не показывать все advanced-поля сразу.
- Не помечать неиспользуемые local servers как ошибку.
- Не выводить secrets в UI или diagnostic report.
- Не обещать автоматическую проверку stealth, если фактически нужна ручная проверка через screen-share preview.
- Не смешивать диагностику с изменением настроек без явного действия пользователя.

---

## Вопросы для решения перед реализацией

1. Делать wizard отдельным окном или специальным режимом Settings?
2. Показывать wizard автоматически только при отсутствии конфига или также после крупных обновлений?
3. Нужна ли кнопка диагностики прямо на overlay bar или достаточно Settings?
4. Нужно ли хранить дату и результат последней успешной readiness-проверки?
5. Должен ли `Проверить всё` автоматически записывать короткий mic sample или сначала спрашивать подтверждение?
6. Нужен ли отдельный guided step для установки local AI или достаточно перехода в существующую установку?
