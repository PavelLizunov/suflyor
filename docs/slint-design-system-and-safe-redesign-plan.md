# Slint design system and safe redesign plan

## Статус документа

Это проектный каркас перед визуальным редизайном. Он не требует немедленной переделки интерфейса и не описывает конкретный новый стиль.

Цель: сделать так, чтобы внешний вид приложения можно было менять постепенно, декларативно и предсказуемо, не ломая поведение окон, stealth-режим, DPI, размещение тайлов и кликабельность элементов.

## 1. Что уже есть

UI построен на Slint и разбит на отдельные поверхности:

- `ui/overlay_bar.slint` — основная панель;
- `ui/tile.slint` — тайл ответа;
- `ui/settings_panel.slint` — настройки;
- `ui/palette.slint` — палитра команд;
- `ui/text_ask.slint` — текстовый запрос;
- `ui/capture_overlay.slint` — выбор области захвата;
- `ui/wizard.slint` — мастер первого запуска;
- `ui/help.slint` — помощь;
- `ui/replay.slint` — просмотр истории;
- `ui/theme.slint` — цветовые токены четырех тем;
- `ui/index.slint` — публичные экспорты UI-компонентов.

Уже существует правильная база: цвета вынесены в семантический `Theme`, а интерфейс описан декларативно. Эту модель стоит сохранить и усилить.

Текущие зоны роста:

- `settings_panel.slint` уже около 84 KB и будет тяжело редактироваться при редизайне;
- `overlay_bar.slint` и `tile.slint` содержат чувствительные зоны `TouchArea`;
- часть визуальных исключений еще задана локальными цветами и размерами;
- размеры некоторых окон одновременно являются частью UI и частью Win32-логики;
- `Theme` применяется отдельно к каждому экземпляру окна, поэтому новую поверхность легко случайно оставить с неправильной темой.

## 2. Главный принцип

Редизайн должен менять представление, но не владеть поведением окна.

Рекомендуемая граница ответственности:

| Слой | За что отвечает |
| --- | --- |
| `Theme` и `Metrics` | Цвета, размеры, отступы, радиусы, типографика, длительности анимаций |
| Slint-компоненты | Компоновка, визуальные состояния, доступные callback-события |
| Rust orchestration | Открытие и закрытие окон, состояние приложения, данные, команды |
| Win32 helpers | Stealth, HWND, taskbar, DPI, позиционирование, drag, захват экрана |

Нельзя прятать Win32-поведение внутрь визуальных компонентов или менять HWND-геометрию ради косметического эффекта.

## 3. Каркас дизайн-системы

### 3.1. Токены вместо локальных значений

Все повторяемые визуальные решения должны иметь декларативные токены.

Цветовые токены уже есть в `Theme`. Их стоит дополнить токенами размеров, например отдельным глобальным `Metrics`:

| Группа | Примеры |
| --- | --- |
| Spacing | `space-xs`, `space-sm`, `space-md`, `space-lg`, `space-xl` |
| Radius | `radius-control`, `radius-panel`, `radius-window` |
| Control size | `control-sm`, `control-md`, `control-lg`, `icon-button-size` |
| Typography | `font-caption`, `font-body`, `font-label`, `font-heading`, `font-mono` |
| Border | `border-thin`, `border-focus` |
| Motion | `motion-fast`, `motion-normal` |
| Opacity | `surface-opacity`, `muted-opacity`, `disabled-opacity` |

Правила:

- компоненты используют семантические токены, а не случайные числа и цвета;
- сырой hex-цвет вне `theme.slint` допустим только как редкое документированное исключение;
- hover, pressed, disabled, focus, loading и error задаются как явные состояния;
- четыре существующие темы проверяются после изменения любого цветового токена;
- размеры критичных окон не меняются через токен без проверки Rust-констант и Win32-позиционирования.

### 3.2. Семантика цветов

Цвет означает роль, а не конкретный оттенок:

- `bg-base`, `bg-surface`, `bg-elevated`;
- `text-primary`, `text-secondary`, `text-muted`;
- `accent`, `accent-hover`;
- `success`, `warning`, `danger`;
- `border`, `border-strong`;
- `code-bg`, `code-text`, `scrim`.

Для нового состояния сначала выбирается существующая роль. Новый цветовой токен добавляется только если роль действительно новая.

### 3.3. Набор переиспользуемых компонентов

Перед большим редизайном полезно выделить небольшой набор простых компонентов:

- `IconButton`;
- `TextButton`;
- `ToggleRow`;
- `FormRow`;
- `SectionHeader`;
- `StatusBadge`;
- `DiagnosticRow`;
- `WindowFrame`;
- `DragHandle`;
- `EmptyState`;
- `LoadingState`;
- `ErrorState`;
- `Tooltip`.

Не нужен универсальный мегакомпонент со множеством режимов. Компонент стоит выделять, когда он повторяется или несет важное правило: стабильный размер, focus, tooltip, состояние disabled или безопасную зону клика.

### 3.4. Иконки

Для критичных действий нельзя полагаться только на emoji: отображение зависит от шрифта и версии Windows.

Рекомендуемый подход:

- завести единый реестр встроенных иконок;
- использовать знакомые символы для pin, close, copy, send, retry, settings;
- давать tooltip неочевидным иконкам;
- не менять размер кнопки при hover;
- проверять отображение на системах без дополнительных шрифтов;
- emoji оставлять только для некритичных декоративных акцентов.

## 4. Правила визуального языка

Приложение является рабочим desktop-инструментом, а не промостраницей. Интерфейс должен оставаться спокойным, компактным и быстрым для считывания.

### Иерархия

- один главный акцент на поверхности;
- крупный заголовок только там, где он действительно нужен;
- вторичный текст заметно тише основного;
- статусные цвета используются для состояния, а не как украшение;
- длинные тексты уходят в scroll, а не раздвигают окно непредсказуемо.

### Геометрия

- стабильные размеры кнопок, чипов и полей;
- одинаковый тип элемента имеет одинаковую высоту;
- hover не должен менять компоновку;
- радиусы ограничены небольшим набором токенов;
- избегать вложенных карточек и декоративных контейнеров без функциональной причины;
- прозрачность применяется к оболочке осознанно и не ухудшает контраст текста.

### Текст

- проверять русский и английский текст;
- не рассчитывать на короткую длину подписи;
- не использовать отрицательный `letter-spacing`;
- не масштабировать шрифт от ширины окна;
- важная подпись не должна исчезать из-за обрезки без tooltip или альтернативного способа прочитать ее;
- для markdown отдельно проверять длинные URL, code block и таблицы.

### Состояния

Для каждой интерактивной поверхности должны быть предусмотрены:

- empty;
- loading;
- streaming;
- success;
- warning;
- error;
- disabled;
- offline или server unavailable, если поверхность зависит от сервера.

## 5. Технические контракты, которые нельзя сломать дизайном

### 5.1. Новое окно

Любое новое Slint-окно должно пройти один и тот же чек-лист:

- экспортировано через `ui/index.slint`;
- получает текущую тему при создании;
- получает текущий stealth-режим до показа пользователю;
- не появляется во время screen share короткой вспышкой;
- корректно получает `skip taskbar`, если окно вспомогательное;
- имеет предсказуемые close и Escape;
- корректно ведет себя с focus;
- проверено на нескольких DPI;
- не мешает drag и кликам соседних элементов.

### 5.2. Stealth и показ окон

В приложении окно сначала должно получить HWND, Win32-декорацию и флаг исключения из захвата, и только затем появиться в видимой позиции.

Дизайн-изменение не должно обходить существующий stealth-aware путь показа окон. Особенно это важно для настроек, палитры, help, wizard и новых вспомогательных окон.

### 5.3. Размеры тайла

Размеры `TileWindow` связаны с Rust-константами размещения. Если меняются `preferred-width`, `preferred-height`, `min-width` или `min-height`, требуется синхронная проверка:

- дефолтных размеров в Rust;
- размещения нескольких тайлов;
- maximize и restore;
- ограничения по рабочей области монитора;
- mixed-DPI конфигурации;
- поведения длинного markdown.

Win32 должен размещать окно, а Slint — задавать его естественный DPI-aware размер.

### 5.4. `TouchArea`

Это одна из самых чувствительных частей UI.

Правила:

- drag-зона не накрывает интерактивные кнопки;
- фоновый `TouchArea` не перекрывает чипы и поля;
- зона клика имеет фиксированный размер;
- после drag кнопки остаются кликабельными;
- каждое изменение порядка слоев проверяется вручную;
- визуально пустой прямоугольник тоже считается подозрительным, если у него есть `TouchArea`.

### 5.5. Capture overlay

Экран выбора области захвата является отдельным критичным контуром:

- покрывает весь виртуальный desktop;
- проверяется на двух мониторах;
- корректно работает с отрицательными координатами второго монитора;
- не получает случайные отступы, рамки и ограничения обычных окон;
- остается совместимым со stealth-режимом.

## 6. Риск-профиль поверхностей

| Поверхность | Риск редизайна | Причина |
| --- | --- | --- |
| Capture overlay | Критичный | Виртуальный desktop, выделение области, multi-monitor |
| Overlay bar | Высокий | Drag, click-through, чипы, HWND, компактная геометрия |
| Tile | Высокий | Markdown, drag, кнопки, resize, maximize, placement |
| Settings | Средний | Много секций, тема, scroll, локальные исключения |
| Palette | Средний | Focus, клавиатура, результаты поиска |
| Text ask | Средний | Focus, ввод, отправка, закрытие |
| Wizard | Средний | Первое впечатление, последовательность шагов |
| Help | Низкий | В основном контент и навигация |
| Replay | Низкий | История и чтение данных |

Высокий риск не означает запрет на редизайн. Это означает более строгую проверку после каждого небольшого изменения.

## 7. Что подготовить до визуального редизайна

### Этап 0. Зафиксировать эталон

- сделать скриншоты каждой поверхности;
- сохранить основные состояния: empty, loading, error, filled, streaming;
- отдельно сохранить тайл с длинным markdown, code block и таблицей;
- зафиксировать вид четырех тем;
- зафиксировать вид при масштабе Windows 100%, 125% и 150%;
- проверить один и два монитора.

### Этап 1. Упорядочить токены без изменения внешнего вида

- вынести повторяемые размеры в `Metrics`;
- убрать необязательные локальные hex-цвета;
- назвать токены по роли;
- оставить внешний вид максимально прежним;
- проверить все четыре темы.

### Этап 2. Выделить примитивы без изменения внешнего вида

- вынести повторяемые кнопки, статусы, строки формы и оконную оболочку;
- закрепить размеры интерактивных зон;
- централизовать tooltip;
- сохранить callbacks и поведение неизменными.

### Этап 3. Разделить крупные UI-файлы

В первую очередь разделить `settings_panel.slint` на оболочку и вкладки. Основной файл должен отвечать за навигацию и переключение секций, а каждая вкладка — за свою разметку.

При необходимости затем аккуратно выделить части `overlay_bar.slint` и `tile.slint`, но не переносить одновременно визуальный стиль и поведение.

### Этап 4. Менять внешний вид по поверхностям

Рекомендуемый порядок:

1. Help и replay;
2. wizard;
3. settings;
4. palette и text ask;
5. overlay bar;
6. tile;
7. capture overlay только при явной необходимости.

Так сначала стабилизируется дизайн-язык на менее рискованных поверхностях.

### Этап 5. Финальная полировка

- унифицировать иконки;
- проверить tooltip;
- проверить contrast;
- проверить клавиатурную навигацию;
- проверить русский и английский текст;
- повторить screen-share тест.

## 8. Матрица визуальной проверки

Минимальная матрица после заметного изменения:

| Измерение | Набор |
| --- | --- |
| Theme | Glacier, Graphite, Obsidian, Light Frost |
| Язык | RU, EN |
| DPI | 100%, 125%, 150% |
| Мониторы | Один монитор, два монитора |
| Stealth | Off, On |
| Данные | Empty, loading, error, filled, streaming |
| Markdown | Обычный текст, длинный текст, code block, таблица, длинный URL |
| Управление | Click, hover, drag, Escape, focus, keyboard input |

Для небольшого изменения можно использовать сокращенный smoke-набор. Перед релизом проходится полная матрица.

## 9. Definition of done для дизайн-каркаса

До начала активного редизайна желательно выполнить следующее:

- цвета и размеры имеют единые декларативные токены;
- новые окна создаются по одному документированному шаблону;
- повторяемые controls выделены в простые компоненты;
- `settings_panel.slint` разделен на вкладки;
- есть эталонные скриншоты основных поверхностей;
- зафиксирована визуальная тест-матрица;
- существует ручная проверка screen-share stealth;
- существует ручная проверка mixed-DPI и двух мониторов;
- Slint attribution остается доступным из приложения в соответствии с лицензией.

## 10. Чек-лист для каждого дизайн-изменения

- [ ] Изменение относится только к одному понятному визуальному слою.
- [ ] Новые цвета и размеры заведены как токены.
- [ ] Hover и pressed не меняют геометрию.
- [ ] `TouchArea` не перекрывает соседние controls.
- [ ] Русский и английский текст помещаются.
- [ ] Проверены Light Frost и хотя бы одна темная тема.
- [ ] Проверены 100% и 150% DPI.
- [ ] Для bar или tile проверены click, drag и focus.
- [ ] Для нового окна проверены theme, stealth и skip-taskbar.
- [ ] Для изменения размеров tile синхронизированы Rust-константы и размещение.
- [ ] Проверен screen-share preview при включенном stealth.
- [ ] Изменение не смешано с рефакторингом поведения без отдельной причины.

## 11. Отдельные решения, которые стоит принять перед редизайном

1. Какая плотность интерфейса нужна: очень компактная или умеренно компактная.
2. Остаются ли все четыре темы или часть из них лучше убрать.
3. Нужен ли встроенный набор иконок вместо системных glyph и emoji.
4. Как хранить эталонные скриншоты: в репозитории или как release-artifacts.
5. Какой минимальный набор accessibility-проверок обязателен перед релизом.
6. Какие размеры overlay bar и tile считаются продуктовым контрактом и меняются только осознанно.

## 12. Итоговая рекомендация

Сначала стоит сделать невидимый для пользователя дизайн-каркас: токены, примитивы, разделение крупных файлов, эталонные скриншоты и чек-листы. После этого внешний вид можно менять поверхностно и поэтапно, не затрагивая фундамент приложения.

Для этого проекта лучший ориентир — декларативная Slint-дизайн-система с небольшим числом понятных компонентов и строгим контрактом между UI, Rust orchestration и Win32.
